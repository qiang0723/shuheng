# Q3 · explore_reader_prices.is_st 推导规则（2026-07-07）

> Q3 归一列 `is_st`（当日是否处于风险警示/ST 状态，PIT）的口径文档，随 Q3 验收包交人过目。载体视图 = `explore_reader_prices`（`qbase/sql/008_explore_reader.sql`）。
> 铁律 4/7：含退市宇宙、PIT 归一。is_st 必须是**历史时点的真实状态**（当日这只票是否 ST），不得用最新名回填。

## 1. 源

`entity_alias`（`alias_type='name'`，忠实存全的 tushare `namechange`，batch7，20005 行）。ST 状态由名称是否含「ST」体现（含 `*ST`）。
⚠ namechange 源系统性脏（Q1 已记：全宇宙 18% 同自然键碰撞——end_date 空/填、错别字同日两名等），Q1 人裁「忠实存全、归一留 Q3 视图」，即脏数据的归一是本视图的职责。

## 2. 缺陷背景（骗不了人，原样记录）

**旧逻辑（008 初版）**：`is_st = EXISTS(存在含ST的 name 行，其 [start_date, end_date] 覆盖当日；end_date IS NULL 当作至今有效)`。
**缺陷**：namechange 对同一改名事件常并存两行——一行 end_date 正确、一行 **end_date=NULL**。NULL-end 的含 ST 行把一个**已结束的** ST 区间永久拉成「至今有效」，导致该票**摘帽后仍恒判 is_st=true**。
**实测发现（2026-07-07）**：`000004.SZ`（多次 ST↔摘帽）2024-01-02 当日真名「国华网安」（已摘帽），旧视图判 is_st=true。
**范围量化**（查 entity_alias）：1102 票有 NULL-end 含 ST 行，其中 **446 票**后来真摘帽 → 摘帽后被误判恒 ST，会污染切片3 的 ST 剔除/分层。

## 3. 主逻辑 = latest-effective-name PIT（我方施工设计，报否决权）

不再信任每行 end_date，改用「下一段起点」划界：

```sql
name_seg AS (   -- namechange PIT 段
  SELECT ts_code, start_date,
         bool_or(alias LIKE '%ST%') AS seg_has_st,        -- 段任一含 ST(含 *ST)
         bool_or(alias LIKE '%退')  AS seg_delist_clean,  -- 退市整理期 'XX退'
         LEAD(start_date) OVER (PARTITION BY ts_code ORDER BY start_date) AS next_start
  FROM entity_alias WHERE alias_type='name' AND batch_id=<max namechange>
  GROUP BY ts_code, start_date                            -- ← 折叠同 start 孪生行,免疫 NULL end_date
)
-- 当日段 = start_date <= trade_date < next_start(段不重叠,至多命中一段)
is_st = CASE WHEN seg_delist_clean THEN false             -- ② 退市整理期字面 false
             WHEN seg_has_st       THEN true              -- ① 段任一含 ST
             ELSE false END                               -- 无 namechange/早于首段 → false
```
两个关键：① `GROUP BY (ts_code,start_date)` 折叠 NULL/非NULL end 孪生为一段 → 段边界只由 `LEAD` 决定，NULL end_date 失去杀伤力（修 446 票污染）；② `'%ST%'` 天然覆盖 `'*ST'`（`*ST深南` LIKE `%ST%` 为真）。

## 4. 边缘脏点裁定（人 2026-07-07，原文即口径，拆分判）

- **① 同日双名冲突（`000995.SZ` 类）= 保守**：任一名含 ST（含 *ST）即 is_st=true；过渡日一天粒度，宁多判勿漏，规则确定可复现。
- **② 'XX退' 退市整理期（`002417.SZ` 类）= 字面 → is_st=false**。**依据是制度事实而非字面偏好**：整理期限幅=10%（首日不设限），非 5%；判 ST 会令 limit_status 拿 5% 算板价、真实 10% 触板反而漏检，分层报告也会把 10% 删失强度错并入 5% 层；字面判恰好落回主板 10%，数值正确。
- **③ 否三态方案**：不为两只票的边角给切片3 加 st_uncertain 三态复杂度。

## 5. 两条已知近似（入本文档，不修）

1. **退市整理期首日无限幅**（334 票 × 首日 1 行，人裁不修）：整理期首日交易所不设涨跌停，limit_status 仍按 board 10% 判，首日 1 行可能误标。见 limit_status 规则文档 §6。
2. **事件结构上不落整理期**：淘沙事件（业绩预告 #4/#5）在结构上不发生于退市整理期；估计窗若触及整理期，由覆盖门槛（160 日内有效交易日 ≥112）与板块分层自然暴露/剔除，无需专门处理。
